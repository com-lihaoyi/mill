= Spring Boot, Micronaut and Quarkus with Mill
// tag::header[]
:author: Vasilis Nicolaou
:revdate: March 2026

_{author}, {revdate}_

Modern Java development is increasingly shaped by frameworks that optimise for cloud-native workloads, fast startup and developer productivity. https://spring.io/projects/spring-boot[Spring Boot], https://micronaut.io/[Micronaut] and https://quarkus.io/[Quarkus] represent three dominant approaches: from Spring's mature and expansive ecosystem, to Micronaut's and Quarkus' focus on performance and native images as a way of building microservices.

// end::header[]

In this post, we explore how the Mill build tool supports these frameworks, providing a faster and smoother developer experience than
traditional build tools like Maven and Gradle.

== Why use Mill?

Mill offers xref:mill::index.adoc#_simplicity[simplicity] for both getting started and extending its capabilities, which means great user experience for
starting of a project, fresh setup of an existing project and writing extensions.

The latter is also clear everytime we extend Mill for every framework, whether Android, Quarkus, Micronaut or Spring Boot.
Extending Mill feels natural, with minimal to no weird workarounds in the code.

Mill is also xref:mill::index.adoc#_performance[superior in performance] when measuring compile/build times.

== Overview

Supporting modern Java frameworks is more than adding dependencies and constructing classpaths. It requires understanding how each framework models:

- Dependency resolution
- Build-time vs runtime behavior
- Code generation (e.g. via annotation processors) and Ahead Of Time (AOT) processing
- Native image integration

Spring Boot remains the https://www.geeksforgeeks.org/blogs/top-java-frameworks/[most popular choice] for enterprise Java due to its extensive tooling and integrations, while Micronaut and Quarkus push the ecosystem forward with innovations like build-time processing, low memory footprint and fast startup times.

Each framework takes different approaches in these areas which affect how build tools integrate with them.
Supporting all three is therefore a good test of the versatility of Mill's own design.

=== Technical details

In this article, we compare the differences between the frameworks and what it means for Mill to support them, implementation wise. Mainly we'll talk about 4 categories.

==== Dependency Management

Most frameworks rely on curating versions of certain libraries by a Bill Of Materials (BOM). This is true for
all the frameworks mentioned in this article.

==== Build Philosophy

Then, we compare the compilation approach of the source code and dependencies. For example, Spring has traditionally relied on *Runtime* and reflection-based dependency injection mechanisms. However, it also provides
a utility for optimising its post-compilation stage with what is called *Ahead Of Time (AOT)* compilation. This turns a lot of reflection-based dynamic invocations, into compile time. This mechanism also exists in Micronaut.

Quarkus is different, and it has its own bootstrapping together. Thus, a build tool would need to delegate a lot of the build flow to Quarkus Bootstrap. In addition, it has its own dependency resolution, which is optimised for native images. While Micronaut and Spring rely on https://github.com/oracle/graalvm-reachability-metadata[reachability-metadata] for building native images with *GraalVM*, Quarkus has drop-in
deployment dependencies ready to be used for native, by replacing certain runtime dependencies.


=== High-Level Comparison

|===
|Framework | Build-Time Behavior | Native Support

|Spring Boot
|Optional AOT
|Supported via AOT + GraalVM

|Micronaut
|Optional AOT
|Supported via AOT + GraalVM

|Quarkus
|Build-time augmentation, built-in optimisations
|Built-in bootstrapping
|===

== Spring Boot Support in Mill

Spring Boot is the most "mainstream" of the three from a build tool's perspective. Getting a Spring Boot application
with AOT and native build targets is as simple as:

[code,scala]
----
package build

import mill.*
import mill.javalib.*
import mill.javalib.spring.boot.SpringBootModule

object `package` extends SpringBootModule, MavenModule {

  override def springBootPlatformVersion = "3.5.7"

  override def artifactName: T[String] = "spring-boot-native-demo"

  object prod extends SpringBootOptimisedBuildModule, MavenModule

  object native extends MavenModule, NativeSpringBootBuildModule {
    def jvmVersion = "graalvm-community:23.0.1"

    // Suppress warning about -H:IncludeResources
    override def nativeImageOptions = Task {
      Seq(
        "-H:+UnlockExperimentalVMOptions"
      ) ++ super.nativeImageOptions()
    }
  }

  def mvnDeps = Seq(mvn"org.springframework.boot:spring-boot-starter-web")

  object test extends SpringBootTestsModule, MavenTests, TestModule.Junit5 {

    def mvnDeps =
      Seq(mvn"org.springframework.boot:spring-boot-starter-test")
  }
}
----

Mill also supports xref:mill::kotlinlib/web-examples.adoc#_spring_boot_aot[SpringBoot with Kotlin], including native as well.

=== Key Observations

- Uses BOM-based dependency management
- AOT is optional and layered on top
- Still largely runtime-driven (compared to Micronaut/Quarkus)
- Building native binaries is a multi-stage process

A Spring Boot app can be built, run and tested by constructing the classpaths from source code and dependencies. From there, a separate compilation process can take the resulting artifacts and optimise the code (e.g. replacing reflection calls with direct method calls). That optimised code can then go through another compilation stage that produces a native binary.

Mill integrates Spring Boot via a `SpringBootModule`, which:

- Injects the Spring BOM automatically
- Locates the main application class dynamically
- Runs Spring’s AOT processor as an external step

The native build is configured via the `NativeImageModule`, which takes care of things like reachability-metadata resolution based on the dependencies.

=== Challenges

- Treating AOT as an optional optimisation
- Managing generated sources across module boundaries

Overall, Spring Boot fits well into Mill’s model because it doesn’t fundamentally change how dependencies or the build flow work. The main difference is giving the choice for AOT + native.

== Micronaut Support in Mill

Micronaut shifts more responsibility to build-time. A Micronaut Mill config can look like this:

[code,yaml]
----
extends: [MavenModule, mill.javalib.micronaut.MicronautNativeAotModule]
micronautPackage: hello.world
bomMvnDeps:
- io.micronaut.platform:micronaut-platform:4.10.3

mvnDeps:
- io.micronaut:micronaut-http-client
- io.micronaut.serde:micronaut-serde-jackson

jvmVersion: graalvm-community:21.0.2
finalMainClass: hello.world.Application
annotationProcessorsMvnDeps:
- io.micronaut:micronaut-http-validation
- io.micronaut.serde:micronaut-serde-processor

annotationProcessorsJavacOptions: !append
- -Amicronaut.processing.incremental=true
- -Amicronaut.processing.annotations=hello.world.*

javacOptions: !append
- -parameters

compileMvnDeps:
- io.micronaut:micronaut-http-client

runMvnDeps:
- io.micronaut:micronaut-http-server-netty
- ch.qos.logback:logback-classic
----

Mill also supports xref:mill::kotlinlib/web-examples.adoc#_micronaut_app[Micronaut with Kotlin].

Another difference (more on the implementation side) is that Spring Boot’s classpath contains the AOT program and relies on the runtime classpath of the application. On the other hand, Micronaut’s AOT utility is an external tool that needs to be resolved and invoked.

=== Key Observations

- Compile-time dependency injection
- External AOT CLI

Mill integrates Micronaut optimisations through two AOT modules, one for JVM and one for native: `MicronautAotModule` and `MicronautNativeAotModule`.

From a build tool's perspective, it remains very similar to the Spring Boot setup, with the main difference being the AOT utility.

=== Native Image Integration

Micronaut extends naturally into native builds:

- Adjusts https://micronaut-projects.github.io/micronaut-aot/latest/guide/configurationreference.html[AOT config] for native runtime
- Adds reachability metadata into GraalVM build inputs

Check out this xref:mill::javalib/web-examples.adoc#_micronaut_native_graalvm[native example] to get a feel for how things are connected.

=== Challenges

- Treating AOT as a first-class build step
- Orchestrating external CLI tools
- Managing generated outputs and classpaths

Compared to Spring, Micronaut is much more build-driven, but still relatively modular in its integration. Both rely on a utility program for AOT, whose output—along with the reachability metadata—is used for building the native image of the app.

For Mill, this use case relies heavily on the correct behaviour of the `NativeImageModule`. Along with resolving dependencies, it also resolves the relevant reachability metadata. As a result, the work to support Micronaut Native also https://github.com/com-lihaoyi/mill/pull/6547[extended the support] for Mill's `NativeImageModule`.

== Quarkus Support in Mill

Quarkus is fundamentally different and, as a result, more complex to support.

Despite the complexity on Mill's side in supporting Quarkus, you can easily configure a Quarkus app with Mill as shown below:

[code, scala]
----
package build

import mill.*, javalib.*

import mill.javalib.quarkus.QuarkusModule

object `package` extends QuarkusModule, MavenModule {

  def jvmVersion = "graalvm-community:23.0.1"
  def quarkusPlatformVersion = "3.31.3"

  def artifactGroupId = "com.lihaoyi.example"
  def artifactId = "7-quarkus-todo"
  def artifactVersion = "0.0.1-SNAPSHOT"

  def mvnDeps = Seq(
    mvn"io.quarkus:quarkus-arc",
    mvn"io.quarkus:quarkus-rest",
    mvn"io.quarkus:quarkus-qute",
    mvn"io.quarkus:quarkus-rest-qute",
    mvn"io.quarkus:quarkus-jdbc-h2",
    mvn"io.quarkus:quarkus-hibernate-orm-panache"
  )

  trait QuarkusTestBase extends MavenTests, QuarkusJunit {
    def mvnDeps = Seq(
      mvn"io.rest-assured:rest-assured"
    )
  }

  object test extends QuarkusTestBase

  object `native-test` extends QuarkusTestBase, QuarkusNativeTest {
    def moduleDeps = super.moduleDeps ++ Seq(test)
  }
}
----

You can try more non-trivial examples xref:mill::javalib/web-examples.adoc#_quarkus_todomvc_example[here]

=== Key Observations

- Build-time augmentation is core
- Requires a full "application model"
- Separates runtime and deployment dependencies

Unlike Spring or Micronaut, Quarkus does not simply run your classpath—it rebuilds it.

Quarkus itself provides internal support for https://github.com/quarkusio/quarkus/tree/main/devtools/maven[Maven] and https://github.com/quarkusio/quarkus/tree/main/devtools/gradle[Gradle]. While these can be followed up to a degree to understand how a build tool should integrate with Quarkus, the difference in approach still acts as a stress test for Mill's design (though not as much as xref:blog::15-android-build-flow.adoc[Android]).

The main challenge is modelling dependencies in the way Quarkus expects them in its application model. Each dependency can be a compile, runtime or deployment dependency.

There are two important nuances with dependencies:

First, deployment dependencies are not declared or resolved in the "traditional" way. Instead, they act as drop-in counterparts to runtime dependencies. This appears to be Quarkus’ way of curating dependencies for native builds, rather than relying on the https://github.com/oracle/graalvm-reachability-metadata[reachability metadata repository].

Second, extension dependencies are derived from runtime dependencies that contain a descriptor file inside the JAR under `META-INF`. These need to be detected and flagged correctly.

Since Mill uses coursier for dependency resolution, all logic for resolving dependencies, finding deployment dependencies and flagging extensions builds on top of the existing resolution pipeline.

=== The Application Model

Mill must construct a Quarkus Application Model before anything can run:

[source,scala]
----
def quarkusAppModel: T[ApplicationModelWorker.AppModel]
----

This model includes:

- Dependencies (runtime, deployment, compile)
- Module metadata (sources, resources)
- Platform BOMs
- Build directories

It is then serialized and passed into Quarkus bootstrap:

[source,scala]
----
ApplicationModelSerializer.serialize(/*...*/)
----

=== Dependency Classification

One of the hardest parts is dependency classification:

- Runtime dependencies
- Deployment dependencies (paired artifacts)
- Extension detection via META-INF

[source,scala]
----
runtimeDeps.filter(hasDeploymentDep)
----

This requires:

- Inspecting JAR contents
- Detecting Quarkus extension descriptors
- Expanding dependencies dynamically

=== Build Execution

Once the model is built, Quarkus takes over:

[source,scala]
----
val augmentAction = quarkusBootstrap.bootstrap().createAugmentor()
val app = augmentAction.createProductionApplication()
----

This produces:

- quarkus-run.jar
- Optional native binary

=== Mill Integration Challenges

- Reconstructing Quarkus’ internal model from scratch
- Mimicking Gradle/Maven behaviour without those tools
- Handling deployment vs runtime classpaths
- Serializing/deserializing build state

Quarkus effectively owns the build, and Mill must adapt to it.

== Conclusion

Supporting Spring Boot, Micronaut and Quarkus in Mill is less about adding features and more about adapting to three different philosophies of Java development.

- Spring Boot fits naturally into Mill’s existing model
- Micronaut extends it with structured AOT processing
- Quarkus requires a deeper integration that reshapes how builds are understood

Together, they illustrate the evolution of Java—from runtime-heavy frameworks to optimised and native systems. Build tools must continuously solve these challenges to keep up.
