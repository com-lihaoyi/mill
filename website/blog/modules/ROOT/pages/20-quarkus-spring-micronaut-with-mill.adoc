= Spring Boot, Micronaut and Quarkus with Mill
// tag::header[]
:author: Vasilis Nicolaou
:revdate: March 2026

_{author}, {revdate}_

Modern Java development is increasingly shaped by frameworks that optimize for cloud-native workloads, fast startup and developer productivity. https://spring.io/projects/spring-boot[Spring Boot], https://micronaut.io/[Micronaut] and https://quarkus.io/[Quarkus] represent three dominant approaches: from Spring's mature and expansive ecosystem, to Micronaut's and Quarkus' focus on performance and native images as a way of building microservices.

// end::header[]

Spring Boot remains the https://www.geeksforgeeks.org/blogs/top-java-frameworks/[most popular choice] for enterprise Java due to its extensive tooling and integrations, while Micronaut and Quarkus push the ecosystem forward with innovations like build-time processing, low memory footprint and fast startup times.

In this post, we explore these frameworks from a build tool implementation standpoint, and more specifically, Mill's.

== Motivation

Mill aims to be a simple, composable and scalable build tool. Supporting modern Java frameworks is more than adding dependencies and constructing classpaths. It requires understanding how each framework models:

- Dependency resolution
- Build-time vs runtime behavior
- Code generation (e.g. via annotation processors) and Ahead Of Time (AOT) processing
- Native image integration

Spring Boot, Micronaut and Quarkus each take fundamentally different approaches in these areas. Supporting all three is therefore a good test of the versatility of Mill's own design.

== Spring Boot Support in Mill

Spring Boot is the most "mainstream" of the three from a build tool's perspective.

=== Key Observations

- Uses BOM-based dependency management
- AOT is optional and layered on top
- Still largely runtime-driven (compared to Micronaut/Quarkus)
- Building native binaries is a multi-stage process

A Spring Boot app can be built, run and tested by constructing the classpaths from source code and dependencies. From there, a separate compilation process can take the resulting artifacts and optimise the code (e.g. replacing reflection calls with direct method calls). That optimised code can then go through another compilation stage that produces a native binary.

Mill integrates Spring Boot via a `SpringBootModule`, which:

- Injects the Spring BOM automatically
- Locates the main application class dynamically
- Runs Spring’s AOT processor as an external step

The native build is configured via the `NativeImageModule`, which takes care of things like reachability-metadata resolution based on the dependencies.

A Mill configuration example on how to setup a Spring Boot project with all three (JVM, AOT and Native) can be found xref:mill::javalib/spring-boot.adoc#_spring_boot_with_aot_processing_and_native_image_support[here].

=== Challenges

- Treating AOT as an optional optimization
- Managing generated sources across module boundaries

Overall, Spring Boot fits well into Mill’s model because it doesn’t fundamentally change how dependencies or the build flow work. The main difference is giving the choice for AOT + native.

== Micronaut Support in Mill

Micronaut shifts more responsibility to build-time. However, from a build tool perspective, it remains very similar to the Spring Boot setup, with the main difference being the AOT utility.

Another difference (more on the implementation side) is that Spring Boot’s classpath contains the AOT program and relies on the runtime classpath of the application, while Micronaut’s AOT utility is an external tool that needs to be resolved and invoked.

=== Key Observations

- Compile-time dependency injection
- External AOT CLI

Mill integrates Micronaut optimisations through two AOT modules, one for JVM and one for native: `MicronautAotModule` and `MicronautNativeAotModule`.

=== Native Image Integration

Micronaut extends naturally into native builds:

- Adjusts AOT config for native runtime
- Adds reachability metadata into GraalVM build inputs

Check out this xref:mill::javalib/web-examples.adoc#_micronaut_native_graalvm[native example] to get a feel for how things are connected.

=== Challenges

- Treating AOT as a first-class build step
- Orchestrating external CLI tools
- Managing generated outputs and classpaths

Compared to Spring, Micronaut is much more build-driven, but still relatively modular in its integration. Both rely on a utility program for AOT, whose output—along with the reachability metadata—is used for building the native image of the app.

For Mill, this use case relies heavily on the correct behaviour of the `NativeImageModule`. Along with resolving dependencies, it also resolves the relevant reachability metadata. As a result, the work to support Micronaut Native also https://github.com/com-lihaoyi/mill/pull/6547[extended the support] for Mill's `NativeImageModule`.

== Quarkus Support in Mill

Quarkus is fundamentally different and, as a result, more complex to support.

=== Key Observations

- Build-time augmentation is core
- Requires a full "application model"
- Separates runtime and deployment dependencies

Unlike Spring or Micronaut, Quarkus does not simply run your classpath—it rebuilds it.

Quarkus itself provides internal support for https://github.com/quarkusio/quarkus/tree/main/devtools/maven[Maven] and https://github.com/quarkusio/quarkus/tree/main/devtools/gradle[Gradle]. While these can be followed up to a degree to understand how a build tool should integrate with Quarkus, the difference in approach still acts as a stress test for Mill's design (though not as much as xref:blog::15-android-build-flow.adoc[Android]).

The main challenge is modelling dependencies in the way Quarkus expects them in its application model. Each dependency can be a compile, runtime, or deployment dependency.

There are two important nuances with dependencies:

First, deployment dependencies are not declared or resolved in the "traditional" way. Instead, they act as drop-in counterparts to runtime dependencies. This appears to be Quarkus’ way of curating dependencies for native builds, rather than relying on the https://github.com/oracle/graalvm-reachability-metadata[reachability metadata repository].

Second, extension dependencies are derived from runtime dependencies that contain a descriptor file inside the JAR under `META-INF`. These need to be detected and flagged correctly.

Since Mill uses coursier for dependency resolution, all logic for resolving dependencies, finding deployment dependencies and flagging extensions builds on top of the existing resolution pipeline.

=== The Application Model

Mill must construct a Quarkus Application Model before anything can run:

[source,scala]
----
def quarkusAppModel: T[ApplicationModelWorker.AppModel]
----

This model includes:

- Dependencies (runtime, deployment, compile)
- Module metadata (sources, resources)
- Platform BOMs
- Build directories

It is then serialized and passed into Quarkus bootstrap:

[source,scala]
----
ApplicationModelSerializer.serialize(...)
----

=== Dependency Classification

One of the hardest parts is dependency classification:

- Runtime dependencies
- Deployment dependencies (paired artifacts)
- Extension detection via META-INF

[source,scala]
----
runtimeDeps.filter(hasDeploymentDep)
----

This requires:

- Inspecting JAR contents
- Detecting Quarkus extension descriptors
- Expanding dependencies dynamically

=== Build Execution

Once the model is built, Quarkus takes over:

[source,scala]
----
val augmentAction = quarkusBootstrap.bootstrap().createAugmentor()
val app = augmentAction.createProductionApplication()
----

This produces:

- quarkus-run.jar
- Optional native binary

=== Mill Integration Challenges

- Reconstructing Quarkus’ internal model from scratch
- Mimicking Gradle/Maven behaviour without those tools
- Handling deployment vs runtime classpaths
- Serializing/deserializing build state

Quarkus effectively owns the build, and Mill must adapt to it.

Despite the complexity on Mill's side in supporting Quarkus, you can easily configure a Quarkus app with Mill as shown xref:mill::javalib/web-examples.adoc#_quarkus_example[here].

== Conclusion

Supporting Spring Boot, Micronaut and Quarkus in Mill is less about adding features and more about adapting to three different philosophies of Java development.

- Spring Boot fits naturally into Mill’s existing model
- Micronaut extends it with structured AOT processing
- Quarkus requires a deeper integration that reshapes how builds are understood

Together, they illustrate the evolution of Java—from runtime-heavy frameworks to optimized and native systems. Build tools must continuously solve these challenges to keep up.
